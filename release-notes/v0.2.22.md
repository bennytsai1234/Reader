# 0.2.22 (2026-04-24)

這個版本是一次針對書源校驗與閱讀恢復定位的穩定性 patch release。重點是讓大量書源校驗不再被慢來源拖住整批，也讓 scroll 模式重新開書時更穩定回到上次閱讀位置。

## 亮點

- 書源校驗改成固定 worker pool，完成一個就補下一個，不再等待同批最慢來源
- 單一書源加入整體校驗上限，逾時來源會被隔離，避免拖慢整個校驗隊列
- 大量校驗時 UI 更新改為節流，降低重繪壓力
- scroll restore 不再顯示「定位中」遮罩，恢復期間改用安靜背景等待
- 恢復定位不再把尚未解析出的 scroll offset 誤存為 `0.0`，避免重開後跳到章節首
- 使用者開始手動滑動時會取消殘留 restore 任務，避免自動定位覆蓋手動操作

## 詳細更新

### 書源校驗

- 校驗流程從批次 `Future.wait` 改為 6 worker 連續取隊列，慢來源不會卡住空閒 worker
- 輸入書源 URL 會先 trim、去空值並去重，避免重複校驗
- 每個來源新增整體 timeout budget，依搜尋/發現檢查內容推算並設上限
- timeout 或未預期錯誤會記錄為隔離狀態，並防止底層晚回來的任務覆寫結果
- `notifyListeners()` 改為 120ms 節流，降低大量來源校驗時的 UI rebuild 密度
- 取消校驗時停止派發新任務，已在執行的任務完成後再收尾

### 閱讀恢復定位

- restore 期間不再顯示文字 overlay，內容尚未可安全顯示時只保留主題背景色
- scroll restore 完成跳轉後會主動完成 restore navigation，避免 token 殘留
- pending restore 缺少 local offset 時會等待 layout ready 再解析，不再預設成章節首
- 已知過期的 `0.0` offset snapshot 會被清除，降低舊資料造成錯誤復位的機率
- 目標章節仍在載入時會 defer restore；確認無法恢復時才取消 navigation
- 使用者開始滑動會清掉 pending restore 與 restore navigation，避免後續突然跳章
