# Reader x Legado Bug 工作流程

## 目的

- 用於明確 bug、錯誤症狀、crash、失敗測試、回歸或不穩定行為。
- 一般 bug 修復不要重新執行 Codebase Atlas；先用 `reader_legado_index.md` 定位，再進入模組文檔與程式碼。
- `legado` 只作為責任邊界、流程順序與失敗處理參考，不是功能待辦。

## 工作流程

1. 打開 [reader_legado_index.md](reader_legado_index.md)，用症狀選出最可能的主模組。
2. 閱讀該模組的「目標專案目前狀態」、「目標專案上下游」、「目標專案變更入口」、「已知風險」。
3. 進入實際程式碼、測試、log 或重現路徑，確認問題是在主模組內發生，還是由上游輸入造成。
4. 若症狀跨模組，依資料、狀態或行為第一次變錯的位置選主模組；其他模組只當邊界模組，改動保持最小。
5. 只查該模組文檔列出的 `legado` 對應區域，借鏡它的責任切分與防呆點，不查 unrelated feature。
6. 修復集中在主模組；跨模組或邊界改動必須有明確原因。
7. 驗證 bug 已修復，且上下游未回歸。
8. 若結構、責任、API 或主要流程改變，更新受影響的 Atlas 模組文檔。
9. 依交付政策完成：提交並推送。完成驗證後建立聚焦 commit，確認目標 remote/branch 後推送；若 remote 或 branch 不清楚，推送前必須先詢問。

## 編輯前摘要

當使用者或專案規則期待確認時，先摘要並等待確認：

- 症狀與重現條件
- 根因假設或已確認根因
- 主模組與邊界模組
- 預計修改檔案
- 修復前後行為，包含使用者可見行為與內部責任切分
- 跨模組改動理由
- 要執行的測試或檢查

## 交付政策

提交並推送。完成驗證後建立聚焦 commit，確認目標 remote/branch 後推送；若 remote 或 branch 不清楚，推送前必須先詢問。
