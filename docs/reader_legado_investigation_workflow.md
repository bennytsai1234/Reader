# Reader x Legado 調查工作流程

## 目的

- 用於行為、歸屬、可行性或根因不清楚，第一步是理解系統的任務。
- 除非使用者明確要求修復或實作，調查流程不直接修改程式碼。
- 一般調查不要重新執行 Codebase Atlas；使用本工作流程與模組文檔即可。

## 工作流程

1. 保留使用者原始問題、症狀、假設或要回答的決策。
2. 打開 [reader_legado_index.md](reader_legado_index.md)，選出最可能的主模組或邊界。
3. 閱讀相關模組文檔的目前狀態、上下游、參考對應、變更入口與已知風險。
4. 檢查必要的程式碼、測試、文件、log、config 或 runtime 路徑，驗證實際行為。
5. 若參考資料有幫助，只看模組文檔列出的 `legado` 對應位置，用來比較責任邊界與 failure handling。
6. 分開記錄已確認事實、假設、未知與限制，不把未驗證推論寫成結論。
7. 追到足以判斷 owner、失敗點或決策邊界後，建議下一個工作流程：bug、優化、重構或驗證。
8. 若調查造成文件更新，依交付政策完成；若沒有檔案變更，摘要調查結果並說明不需要 commit。

## 調查摘要

結束時摘要：

- 原始問題或症狀
- 主模組或邊界
- 已確認事實
- 仍未知事項
- 風險或脆弱假設
- 建議下一步

## 交付政策

提交並推送。若有檔案變更，完成驗證後建立聚焦 commit，確認目標 remote/branch 後推送；若 remote 或 branch 不清楚，推送前必須先詢問。若沒有檔案變更，摘要結果並說明不需要 commit。
